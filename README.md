# Typing Vocab / 타자 연습 도구

打字背单词工具，灵感来自 [metrotyping.kr](https://metrotyping.kr)（韩国地铁打字游戏）的交互形式：
当前字符消失只留下你实际打出的内容，逐部件（초성/중성/종성）高亮对错，配合小提示和进度统计。

在线地址：https://typing-vocab.pages.dev

## 核心架构：不依赖浏览器输入法（IME）

韩语输入不通过浏览器原生的 Hangul 输入法组字（不监听 `compositionstart`/
`compositionend`/`<input>` 的 `value`），而是直接监听物理键盘的 `keydown` 事件
（`event.code`，即物理按键位置），按标准**두벌식（2-set）**键盘映射表自己实现了
一套完整的韩语组字引擎（`makeKoreanEngine`）。

这么做的原因：浏览器原生 Hangul 组字状态和网页脚本对 `<input>` 内容的操作容易
互相脱节（尤其是换词时清空输入框、光标位置被浏览器不可预期地重置），会出现
"打第二个字时内容跑到第一个字位置"的乱序问题。改成自己实现组字引擎后，整个
过程完全在 JS 里确定性地跑（`cho`/`jung`/`jong` 状态机 + `activeIndex` 显式
维护），从根上避开了这类问题。

**这意味着：电脑上使用时不需要把系统切换成韩语输入法**，用英文/QWERTY 物理
键盘直接打字即可，页面会按두벌식规则把物理按键翻译成对应字母（比如 `d`→ㅇ,
`u`→ㅕ, `s`→ㄴ 打出"연"）。手机等触屏设备则通过页面中间的输入框，用系统自带的
韩语/英语软键盘直接输入（`syncWithString`，不走두벌식映射）。

## 界面

配色/字体照着一份 Claude Design 设计稿（`Typing Vocab UI`）重做过：
`Unbounded`（大字标题/数字）+ `Space Grotesk`（正文）。默认深色主题，主色近黑
`#0a0a0a` / 米白 `#f2ede2` / 粉 `#e3b8d6`；头部还有个 🌙/☀️ 开关可以切换成亮色
主题（米白 `#f1efe9` 背景 + 深墨 `#1a1714` 文字 + 浅粉 `#F7CDCE` 主色）。

这套视觉语言现在也抽成了一份独立的 React 组件库（`design-system/`，见下面
"设计系统同步"一节），同步到 claude.ai/design 后可以在那边直接用真实组件继续
设计新界面；`index.html` 本身还是纯 vanilla JS，不依赖这个包，两边靠人工/AI
辅助对齐，不是自动联动。布局分四块：

- **顶部导航栏**（真正的 `position:sticky`，滚多远都贴在顶部）：左边 logo，
  中间한국어/english切换 tab（切换**练习内容**是韩语还是英语词表），中间偏右
  "연습 / 단어 목록 / 단어장 / 통계"四个导航按钮（연습固定是"关闭抽屉回到练习"，
  后三个是开关式——没打开对应页面时点一下展开抽屉，已经打开且正显示那一页时
  再点一下直接收起，细节见下面"单词抽屉"一条），最右边先是一个**🌙/☀️ 主题开关**（暗色/
  亮色切换），再是**KO/EN 界面语言开关**（只翻译界面文案本身，跟练习内容切换
  是两回事，互不影响），两个的选择都记在 localStorage，最后是账号胶囊按钮
  （点开切换账号，旁边✎图标是改名字）。
- **hero 区**：纯装饰，标题+大字叠字，没有交互。
- **练习卡片 + 侧栏**：练习卡片左上角是"当前进度"（如 3/10），右上角三个图标
  按钮（★收藏 / ✎笔记 / ↻重来），中间是实际打字区（罗马音提示 + 音节/字母格 +
  输入框，音节格垂直居中在一个可伸缩的边框区块里），底部是上一个/下一个按钮。
  右边常驻一栏侧栏（不需要打开抽屉就能看到、能操作），现在整体是**一张连续的
  圆角卡片**（不再是几个各自独立的方框），内部用发丝线分成几段：当前单词本条
  （封面+名字+单词数，"⇌ 전환"按钮跳去단어장标签切换单词本，这个不在合并卡片
  里，单独在最上面）、按创建日期筛选的药丸行（见下面"自定义单词本"一节）、
  **"단어 추가"**单独套了一层强调色双边框的子卡片（右上角挂一个"NEW"角标，
  突出这是重点操作）里面是단어输入+뜻输入+"+ 추가"按钮、가져오기/내보내기·
  🤖 AI 프롬프트·초기화三个按钮，最下面是딕테이션 모드开关。**즐겨찾기만/
  노트만/오답만三个筛选和搜索框已经不在侧栏了**——挪到了抽屉里单词列表标签页
  自己的顶部（见下一条），侧栏现在只保留日期筛选和加词/管理相关功能。侧栏
  内容比练习卡片天然要长，所以练习卡片会拉伸到跟侧栏同样高（中间打字区变高、
  内容居中，不是底部露一截空白）。
- **单词抽屉**（从右侧滑出，`min(460px, 92vw)`宽）：**没有关闭×按钮**——顶部
  是"단어 목록/단어장/통계"三个等宽平分的圆角分段 tab（米白底深字表示选中），
  这三个 tab 本身就是开关：没打开时点一下会展开抽屉并跳到对应页面，已经开着
  且正显示这一页时再点一下会直接收起抽屉；顶部导航栏（헤더）里的"단어 목록/
  단어장/통계"三个按钮效果完全一样，也是同一套开关逻辑，跟抽屉里的 tab 保持
  同步。**响应式行为，以视口宽度 900px 为界**：宽屏时抽屉打开**不会盖住**
  左边的练习卡片和侧栏——整个左侧内容会用 `margin-right` 整体向左让出空间，
  抽屉和左边内容同时可见、都能操作，背景也不会变暗；窄屏/手机没有空间并排
  显示，退回传统的全屏浮层——抽屉盖在内容上面，背后有一层半透明黑色遮罩，
  点遮罩空白处、或点顶部导航栏切回"연습"都能关掉。**단어 목록**标签页顶部
  现在固定钉着一张搜索+筛选合并卡片（不随下面列表滚动）：上半是🔍搜索框，
  中间一条发丝线，下半是즐겨찾기만/노트만/오답만三个筛选胶囊；下面才是可
  滚动的单词列表本体，每一行是个圆角大卡片（序号+单词+释义，右侧三个圆形
  图标按钮✎笔记/★收藏/✕删除），当前正在背的那一行会有 accent 描边高亮，
  列表加了滚动吸附（scroll-snap），停下来时永远是整张卡片贴齐在筛选卡片
  下方，不会停在切一半的位置。**단어장**标签页顶部是단어장 검색框 + "+ 추가"
  按钮和一条虚线"+ 새 단어장"整宽按钮，下面不再是方格网格，改成**竖向堆叠的
  马卡龙色圆角卡片列表**（灵感类似 Apple Wallet 卡包）：卡片之间互相压着，
  只露出上面一小截，看起来像一叠票据；鼠标移到某张卡片上它会弹出来（放大+
  上浮+阴影加深），同时上下紧邻的卡片会一起让开一点空隙，方便顺着鼠标继续
  往下扫到下一张而不会被挡住；正在练习的那本常驻比其他卡片略大一圈，一眼
  能看出是当前用的那本。**触屏设备没有鼠标悬停这回事**（`matchMedia('(hover:
  hover)')` 检测），改成滑动时实时判断哪张卡片离滚动区域竖直中心最近，把
  那张当成"选中"的卡片套用同一套放大/让位效果——手指划过去，划到中间的那张
  自动弹出来，逻辑上更接近苹果钱包卡包的感觉；桌面端的悬停逻辑完全不受影响，
  两套逻辑互斥启用，不会同时生效。每张卡片：左上角圆形图标头像，右上角"연습 중"徽章
  （仅当前本子）+ 圆形✎编辑按钮，中间大标题，底部单词数量胶囊 + 圆形→按钮
  （点它进入这个本子）。통계标签页顶部是"복습 현황 / 04 STATS"带分割线的
  标题行，下面四个圆角方块——밀린 복습用纯色填充（accent 粉底+深色数字），
  오늘 복습/예정是细边框卡片，미학습是虚线边框+右上角"NEW"角标——标签统一用
  等宽字体大写+小方块状态点（不再是 emoji），再往下是胶囊形숙련도分布条 +
  취약 단어 TOP5 排行条（有오답的词才会出现）+ 底部活动热力图。

这些都是纯视觉结构，逐部件高亮、SRS、云端存储等判定逻辑不受影响。

## 设计系统同步

`design-system/` 是一份独立的 `@typing-vocab/design-system` React 组件包
（`Button`/`IconButton`/`Badge`/`TextInput`/`Toggle`/`TabGroup`/`Card`/
`ProgressBar`/`CharacterTile`/`PracticeCard`/`Modal`/`AppHeader`），照着
`index.html` 的视觉语言手工提取而成，只给 claude.ai/design 用，**不会被
`index.html` 引用、也不参与生产构建**。同步流程见 `.design-sync/`：

- `.design-sync/config.json`：同步目标 project、构建命令等配置
- `.design-sync/conventions.md`：给设计 agent 看的用法说明（主题切换、
  token 命名、组合方式），会被拼进上传后 project 里的 README
- `.design-sync/previews/*.tsx`：每个组件的手写预览用例
- `.design-sync/NOTES.md`：同步过程中踩过的坑、下次同步要注意的点

日常改 UI 的流程是双轨、异步对齐，不是自动联动：设计新界面在
claude.ai/design 那边的常驻 project 里做（用已同步的真实组件拼装/创新），
定稿后把设计规格（`.dc.html` 模板 + 截图）交给 Claude Code，照着改
`index.html` 里对应的渲染函数；如果这次改动引入了新的通用视觉模式，就顺手
补进 `design-system/src/` 再重新同步一次，下次设计就能直接复用。跑一次
同步大致是：

```bash
cd design-system && npm run build   # 编译组件包
# 见 .design-sync/NOTES.md 里的 "Re-syncs are one command" 一节
```

## 功能

- **韩语模式**：按음절（音节）显示，每个字拆成 초성/중성/종성，逐部件比对：
  - 灰色（浅）：还没轮到的字
  - 深灰色：轮到的字 / 正在打且目前正确
  - 粉色：整个字打完且完全正确
  - 红色：这个字有部分打错了（不影响继续打下一个字），上方会浮现一个小提示
    显示正确写法，直到用 Backspace 改对为止
  - **横杠数按实际物理按键数算**：复合元音（ㅘㅙㅚㅝㅞㅟㅢ）和复合收音
    （ㄳㄵㄶㄺㄻㄼㄽㄾㄿㅀㅄ）在두벌식键盘上都需要按两次键才能拼出来，各占
    2 条杠。收音后面如果又打了元音，收音会正确地挪到下一个字当声母（比如打
    "닭"再打"ㅣ"会变成"달"+"기"）。
  - **无받침的字体号会比有받침的小一点**（34px vs 40px）：即使字号写的一样，
    韩文字形在同一个格子里，没有收音（只有초성+중성 2 个部件）的字天然会比有
    收音（3 个部件）的字占得更满、看起来更大更粗，几个字排在一行时容易显得
    "歪"；按目标音节有没有收音分两档字号后视觉上更统一（字号是按目标结构算的，
    不是按打到哪个部件算，所以打字过程中不会跳动）。
- **英语模式**：逐字母实时比对，每个字母位置独立判断对错（互不影响），
  对了变绿、错了变红。
- `Backspace` 逐个部件/字母删除；打对整个单词后按 `Enter` 进入下一个词，或点
  "다음 단어"跳过；可选"딕테이션 모드"（听写模式）——释义/罗마자提示始终保留
  （不然不知道该背哪个词），但会隐藏字形预览（还没打的字/字母）、屏幕虚拟
  键盘上"下一个该按的键"高亮、韩语打错时的纠正提示，逼自己凭记忆打出拼写，
  而不是看着字形照抄。韩语/英语模式下都支持。
- **方向键翻词**：桌面端按 `←`/`→` 直接切换上一个/下一个单词，跟点
  "◄----"/"----►"按钮效果一致（焦点落在搜索框/笔记等输入框里时不生效，方向键
  照常用来移动光标）。
- **移动端英语输入防错位**：手机上用中文拼音键盘打英语单词时，输入法有时会在
  合成过程中往输入框里插入空格，原来逐字符比对会把这个空格当成一次输入导致
  后面字母整体错位；现在同步前会先过滤掉空白字符，不受这种拼音键盘副作用影响。
- **收藏（즐겨찾기）**：当前词一键收藏，单词列表可只看收藏。
- **笔记（노트）**：给单个词写例句/相关知识，单词列表里有笔记的词会标 📝，并在
  下面附一行笔记内容预览（取前 36 个字符，多行合并成一行，超出用…省略）。
  收藏和笔记都按单词的数据库主键 `id` 单行读写（`PATCH /api/words/:id`），
  不管切换到哪个单词本、日期筛选怎么勾选，只要这个词的 `id` 没变，它的收藏/
  笔记状态在界面上永远实时同步、不会被别的操作覆盖或重置（这是这版重构专门
  修的一个 bug，见下面"词表数据结构"一节）。
- **修改单词（✎）**：单词列表每一行收藏星星前面有个✎按钮，点开可以直接改这
  个词本身的文本和释义/罗马字，不用删了重加。改成的单词文本如果和这个单词本
  里另一个词撞了，会拦下来提示先处理那个词（不会静默生成两个同名词）。
- **搜索**：单词列表顶部搜索框，实时匹配单词、翻译、笔记内容。
- **单词列表定位同步**：点击"단어 목록"（或从단어장/통계标签切回来）时，列表
  会自动滚动到当前正在背的那个词，不用手动往下翻找。
- **自定义单词本（단어장）**：不区分"官方教材"和"自建生词本"——所有单词本都是
  统一的自定义容器，可以自由新建/改名/写简介/删除。단어장标签页顶部有个搜索框
  （实时按名字过滤单词本）和"+ 추가"快捷新建按钮，下面是竖向堆叠的马卡龙色
  圆角卡片（不再是方格网格），卡片之间层叠着、只露出上面一截，鼠标划过会
  弹出来、邻近卡片跟着让开（细节见上面"界面"一节的 Apple Wallet 卡包描述），
  卡片正面是图标+名字+单词数量胶囊，不显示简介，当前正在练习的那本左上角有
  "연습 중"角标、常驻比别的卡片略大，右上角悬浮一个✎编辑按钮（简介还在，只是
  不显示在卡片正面，编辑弹窗里能看到/改）。**新建/编辑单词本弹窗**照一份 Claude
  Design 设计稿重做过：28px 大圆角深色面板，名字输入框留空时边框会变成强调色
  提醒（打字后恢复正常），左边 88px 圆角图标预览挨着右边的图标选择区——图标
  区严格 6 列网格、每格 36px、11px 圆角，选中态是强调色描边+内发光，自定义
  emoji 输入框本身就是网格里最后一格（虚线边框），这块区域限高 150px、超出
  上下滚动，滚动条改成细的透明轨道滑块（`.notebook-emoji-grid` 单独限定，没
  影响全局滚动条样式）；下面"배경색"改成一排**圆形**色块（不再是方格），选中态
  是白色内圈+强调色外圈光晕。封面是预设的**纯色背景 + emoji**（36 个马卡龙色系
  纯色，按色相分 5 组从浅到深排列：暖粉/柔粉 → 梦幻紫/雾霾蓝 → 纯净海盐蓝 →
  冰凉薄荷/青绿 → 温暖黄绿/莫兰迪雅绿，6 列网格铺开；颜色值直接存成
  `coverGradient: "#rrggbb"` 十六进制字符串，不再是渐变名称查表），emoji 统一用
  Apple emoji 图片渲染（走 `cdn.jsdelivr.net/npm/emoji-datasource-apple`，而不是
  系统自带的 emoji 字体），这样 Windows/Android 上看到的图案跟 Mac/iPhone 一致，
  不会因为系统 emoji 字体不同而变形或显示成方块；图片加载失败会自动重试一次
  备用编码，还失败就直接不显示，不会露出裂图图标。后端仍然预留着 `coverUrl`
  字段（这版重做弹窗时按新设计稿去掉了原来"📷 拍照当封面即将上线"那行提示文字，
  但字段本身没变——以后支持上传图片时直接往这个字段存链接即可，不用再改数据库
  结构）。删除单词本会先弹出专门的二次确认弹窗（不是浏览器原生 `confirm()`），
  列清楚会连带删掉几个词、并提示不可撤销；只剩最后一个单词本时禁止删除。单词本
  按한국어/english模式各自独立（一个单词本只属于一种练习语言）。
- **按创建日期筛选（날짜별 필터）**：进入单词本后，筛选行会列出这个本子里所有
  出现过的创建日期，点日期药丸（显示为 MM-DD，鼠标悬停显示完整日期）多选/取消，
  练习区和单词列表就只在选中的那几天导入/添加的词里循环；除了"전체"（全选）
  和逐个日期，还有一个"최근 7일"快捷按钮一键选中最近一周。日期超过 6 个时行
  会横向滚动收起，右上角出现"더보기"展开成多行、"접기"收回去，避免日期一多把
  下面的内容挤没了。加词、批量导入的词都会记下当天日期，不需要手动标记。
  重复粘贴同一批内容不会把已有词变成两行——单词文本判重后的处理见下面
  "加词去重合并"一节，收藏/笔记/SRS 状态始终不受影响。
- **主题切换（🌙/☀️）**：头部账号胶囊左边的独立开关，暗色/亮色两套配色用
  CSS 自定义属性（`--bg`/`--ink-rgb`/`--accent`等）实现，点一下整体切换，
  选择记在 localStorage。**练习卡片和笔记弹窗故意保持奶油色不跟随主题**——
  这两块是设计上刻意保留的"纸感"元素，不是漏改。
- **界面语言（KO/EN）**：顶部账号胶囊左边的独立开关，把导航栏、抽屉、统计
  面板、各类弹窗和提示语整体翻译成英文，练习内容（한국어/english 那个 tab）
  不受影响。是纯静态翻译字典，新增功能时新文案不会自动配上译文，需要手动
  在代码里的 `I18N` 字典里给两种语言各补一条。
- **进度记忆**：当前背到第几个词（按账号+모드+单词本三者分别记录）、最后停留
  在한국어还是english练习模式、最后打开的是哪个单词本，都存在 localStorage
  里。不管是刷新页面、来回切换한국어/english，还是在多个单词本之间来回切换，
  都会各自恢复到上次停留的那个词，不会跳回第一个。
- **AI 提示词**：单词添加区有个"🤖 AI 프롬프트 복사"入口，复制里面写好的提示词
  丢给 AI（配合自己的图片/文档资料），能直接生成本工具批量导入需要的格式。
- **加词去重合并**：不管是单词添加区那个"단어(한국어/english)+뜻/로마자+➕"
  快速加一个词，还是"가져오기/내보내기"粘贴批量导入，只要单词文本这个单词本
  里已经有了（批量导入时同一次粘贴的文本里同一个词出现不止一次也算），都不会
  生成重复的一行——而是把新的뜻用"; "拼接合并进那个词已有的뜻里（一样的뜻不会
  重复拼），该词的收藏/笔记/SRS 状态不受影响。批量导入完成的提示语会分别报
  "新增了几个词"和"合并了几个已有词"。
- **云端存储**：单词、收藏、笔记、历史记录都存在 Cloudflare D1 数据库，不依赖
  浏览器本地存储，换设备、换浏览器、清缓存都不会丢（细节见下文）。
- **间隔重复（통계 탭의 복습 현황）**：每个词有一个 0-5 级的"熟练度"（Leitner
  box）。一次不出错就打对，等级 +1，下次复习间隔拉长（1/3/7/14/30 天）；打
  错了等级 -1，很快又会到期。复习进度按等级/到期日存在词条本身上，跟收藏、
  笔记一样存云端。통계 탭里"복습 현황"下的四个格子——🔴밀린 복습（到期但还没
  复习）/🟡오늘 복습（今天到期）/⚪예정（还没到期）/🆕미학습（从没打过）——
  各自独立可点：点开某个格子就直接把练习区切换成只含那一类词（1/N、2/N 这样
  走，其他类别的词暂时不出现），这个格子是 0 就弹提示告诉你这类没有词；日期
  筛选行里会多出一个"🔁 <类别名> ✕"药丸，点它退出这个筛选、回到完整词表。
  在这个筛选里把一个词完全打对，它的到期日会更新、可能就不再属于原来那个
  类别了，所以列表会跟着实时变短，等这个类别打空了会弹提示并自动退出筛选。
- **正确率**：进度条那行会显示当天的"정확도"（某个词从头到尾一次都没打错就
  算"clean"，clean 词数 / 完成词数），按用户+日期存本地存储，刷新页面不会
  重置，跨天自动清零。
- **错词本（오답만）**：只要某个词在这次作答过程中出现过打错（哪怕后来改对
  了），就会给它的"오답 횟수"+1。平时的单词列表（전체/즐겨찾기만/노트만/搜索）
  不显示这个错误次数，只显示干净的"序号. 单词"，避免`❌N`和序号数字挨在一起
  看着像错位（比如"❌1 56."容易被看成"156."）；勾选"❌ 오답만"才会切换成专门
  的错词列表——按错误次数从多到少排序，每条只显示`❌N 单词`，不带原来的序号
  （排序变了，原序号也没意义了）。

## 使用方法

线上直接打开 https://typing-vocab.pages.dev 即可。首次访问会要求起一个名字，
之后你的数据就按这个名字存在云端；换设备/浏览器时输入**完全一样**的名字（大
小写、空格都算数）才能看到同一份数据。右上角胶囊按钮显示当前账号名，点它是
"계정 전환"（切换去看**另一个**已存在的账号，不会搬数据）；旁边那个✎图标才是
"이름 변경"（把当前账号改个名字，数据不丢，跟切换账号是两回事，别点错）。

本地也可以直接双击打开 `index.html`（无需安装依赖、无需构建），但这样云端
接口访问不到，会用内置默认词表跑在纯本地模式（改动不会保存），适合单纯改
代码时预览。

## 词表数据结构

这版重构之前，`words` 表虽然有数据库 `id`，但前端从来拿不到它——保存永远是
"整表删掉再按数组顺序重新插入"，历史记录回放只能靠**单词文本**做匹配去重。
这就是"点历史记录/切单词本导致收藏和笔记丢失"这个 bug 的根因：数据没有稳定
身份，容易在整表覆盖时被文本重名或顺序错位坑到。

现在前端拿到的每个单词对象都带着数据库主键 `id`，所有写操作（收藏/笔记/SRS/
删除/移动单词本）都是按这个 `id` 的单行 `PATCH`/`DELETE`，不会再有整表覆盖
这种操作，收藏笔记丢失在架构上已经不可能发生。

单词对象结构：韩语 `{id, notebookId, createdDate, word, roman, favorite,
notes, box, dueAt, wrongCount}`，英语把 `roman` 换成 `meaning`，其余字段一样。
单词本对象结构：`{id, name, description, coverGradient, coverEmoji,
coverUrl}`（`coverUrl` 非空时优先用图片，否则用纯色+emoji）。默认词表见
`index.html` 里的 `DEFAULT_KO_WORDS` / `DEFAULT_EN_WORDS`——新账号首次访问时
会真的建一个默认单词本、把这些词批量导入进去存进 D1（不再是像以前那样只停留
在前端内存里，等第一次编辑才顺带存上）。

## 项目结构

```
index.html                      前端全部代码（HTML/CSS/JS 都在一个文件里）
functions/api/notebooks.js      读取某账号某语言模式的单词本列表 / 新建单词本
functions/api/notebooks/[id].js 改名/改简介/改封面（PATCH），删除单词本（DELETE，级联删除本子里的词）
functions/api/words.js          读取某账号某语言模式下所有单词 / 加一个新词（POST）/ 清空某个单词本（DELETE）
functions/api/words/[id].js     按 id 改收藏/笔记/SRS等字段（PATCH）/ 删单个词（DELETE）
functions/api/words/bulk.js     批量导入到某个单词本（新词就新增；单词文本已存在就把新的뜻用"; "合并进那个词原来
                                 的뜻里，不新增一行，也不覆盖那个词的收藏/笔记/SRS 状态）
functions/api/rename.js         把某账号的数据（单词、单词本）整体迁移到一个新名字
schema.sql                      D1 数据库表结构：一份完整的最新 schema，只给全新数据库用
migrations/                     针对"已经部署过的旧数据库"的一次性升级脚本，见下面部署一节
wrangler.toml                   Cloudflare Pages 项目配置（含 D1 数据库绑定）
design-system/                  独立的 React 组件包，只给 claude.ai/design 用，不参与生产构建（见上面"设计系统同步"）
.design-sync/                   design-system → claude.ai/design 的同步配置/预览/踩坑记录
```

## 已知局限

- 目前只支持标准두벌식键盘映射，没有支持其他韩语输入方案（如세벌식）。
- 不设密码——账号完全靠名字区分，知道别人用的名字就能看到/修改对方的数据，
  适合个人及小范围熟人共享，不适合存敏感内容。
- 正确率只是当天的粗略统计，不做历史趋势记录。
- 单词本封面的 emoji 图标依赖外部 CDN（`cdn.jsdelivr.net`）加载 Apple emoji 图片，
  纯离线或该域名被墙的环境下图标会加载不出来（自动隐藏，不会显示裂图，不影响
  其余功能）。

## 云端存储部署（Cloudflare Pages + D1）

当前部署已经接了 GitHub 自动部署——推送到 `main` 分支后 Cloudflare 会自动
重新构建部署，**不需要手动跑部署命令**。以下是从零搭建这套环境的完整步骤，
供换账号/换项目时参考：

```bash
# 1. 登录 Cloudflare（会打开浏览器授权）
npx wrangler login

# 2. 创建 D1 数据库
npx wrangler d1 create typing-vocab-db
# 输出里会有一行 database_id = "xxxxxxxx-xxxx-...", 复制它

# 3. 把 database_id 填进 wrangler.toml
#    打开 wrangler.toml，把 database_id 换成上一步拿到的 id

# 4. 把表结构建到远程数据库
npx wrangler d1 execute typing-vocab-db --remote --file=./schema.sql

# 5. 首次部署（会让你确认创建 Pages 项目）
npx wrangler pages deploy .

# 6. 在 Cloudflare 后台 Workers & Pages → 项目 → Settings → Build，
#    把项目连接到这个 GitHub 仓库，Production branch 选 main，
#    并确认 Settings → Bindings 里有一条 D1 database: DB → typing-vocab-db
#    （连接 Git 之后的自动构建不一定会保留手动部署时的绑定，要检查一下）
```

**如果之前已经建过库**：`schema.sql` 现在是"全新数据库该长什么样"的完整定义，
`CREATE TABLE IF NOT EXISTS`/`CREATE INDEX IF NOT EXISTS` 在已存在的库上重跑是
安全的空操作，但**不会**给已经建好、缺列的旧表补列——补列走 `migrations/` 目录
下的一次性脚本，按需要跑哪个跑哪个（不会丢数据，也不依赖跑没跑过另一个）：

```bash
# words 表里还没有 box/due_at/wrong_count（间隔重复/错词本相关字段）：
npx wrangler d1 execute typing-vocab-db --remote --file=./migrations/001_add_srs_columns.sql

# 还没有 notebooks 表 / words.notebook_id / words.created_date（这版单词本重构加的）：
npx wrangler d1 execute typing-vocab-db --remote --file=./migrations/002_add_notebooks.sql
```

`migrations/002_add_notebooks.sql` 会自动给每个已有 (账号, 语言模式) 建一个
默认单词本，把这个模式下所有词都归进去；每个词的创建日期尽量从旧的
`history_batches` 历史记录里按单词文本匹配回填，匹配不上的记成迁移当天——
过程中只新增/移动，不会删除或覆盖任何词的收藏、笔记、SRS 状态。

**本地开发调试**（不需要真的部署，用本地模拟的数据库测试后端逻辑）：

```bash
npx wrangler d1 execute typing-vocab-db --local --file=./schema.sql
npx wrangler pages dev .
```

如果 `/api/words`、`/api/notebooks` 等接口访问不到（比如还没部署好、或本地直接用
`python -m http.server` 这种纯静态服务器打开），页面会自动降级：用内置的默认
词表跑起来，并在进度条下方显示一行提示，说明这次的改动不会被保存——不会白屏
或报错卡住。

## 故障排查：进度条下面出现红字警告

"클라우드에 연결할 수 없습니다 — 이번 세션의 변경사항은 저장되지 않습니다." 这行红字
代表这次打开页面时 `/api/words`（或 `/api/notebooks`）请求失败了，页面自动降级用
内置默认词表跑。**这不代表数据被删了**——只要这个警告出现，`cloudOk` 就会变
`false`，之后所有单条 `PATCH`/`POST`/`DELETE` 保存操作都会直接跳过（不发请求），
不会拿默认词表覆盖云端的真实数据。原来的数据大概率还老老实实存在 D1 里，只是
这次连不上。

排查顺序：

1. 浏览器开发者工具（`Cmd+Option+I`）→ **Network** 标签，刷新页面，找
   `words?user=...` 这条请求，看**状态码**。
2. 如果是 **500**，Response 里通常是 Cloudflare 自己生成的"Worker threw
   exception"通用错误页，看不出具体原因——去 Cloudflare 后台 Workers & Pages →
   项目 → **Logs**（实时日志）里翻真正的报错。

**已知会导致 500 的原因（都遇到过）：**

- **绑定没跟着部署生效**：在 Cloudflare 后台 Settings → Bindings 里加/改 D1
  绑定，只对**之后**的新部署生效，不会补给已经部署好的旧版本。改完绑定要
  触发一次新部署（随便 push 一个空提交，或去 Deployments 里点 Retry deployment）。
- **迁移语句写错、静默失败**：`ALTER TABLE ... ADD COLUMN` 在 SQLite/D1 里
  **不支持** `IF NOT EXISTS` 修饰符（这个只有 `CREATE TABLE`/`CREATE INDEX`
  才有）。早期版本的 `schema.sql` 把这种 `ALTER TABLE` 语句和 `CREATE TABLE`
  写在同一个文件里，`wrangler d1 execute --file=...` 跑完没有报出显眼的错误，
  但 `ALTER TABLE` 其实一条都没成功执行，代码已经在按新列查询了，于是每次
  请求都 500。**怎么判断迁移是不是真的生效**：看 `wrangler` 跑完打印的统计
  表，`Total queries executed` 要等于文件里的语句条数，涉及改动已有数据的
  语句 `Rows written` 不应该是 0——数字对不上就说明有语句被静默跳过了。
- **`wrangler d1 execute --file` 是把整个文件当一个事务跑的**：只要文件里
  任何一条语句报错（哪怕是"这列已经加过了"这种预期内、本该忽略的报错），
  **整个文件都会回滚**，包括这条报错语句后面那些原本会成功、这次真正需要
  跑的新语句——不是"跳过这一条继续跑下一条"。这也是为什么 `schema.sql` 和
  `migrations/` 要分开：`schema.sql` 只给全新数据库用（每条语句在新库上都
  能成功，不会遇到"已经加过"的情况），针对已部署旧库的 `ALTER TABLE` 全部
  挪到独立的 `migrations/00N_xxx.sql` 里，一个迁移一个文件，不会被另一个
  迁移"这列已经加过"的报错连累回滚。以后新增字段/表时，新的一次性迁移也应
  该继续新开一个 `migrations/00N_xxx.sql`，而不是塞进 `schema.sql` 或者塞进
  前一个迁移文件末尾。

**改完 `schema.sql` 或新加一个 `migrations/*.sql` 时怎么验证：**

1. 先在本地库上跑一遍确认语法没问题，再跑 `--remote`：
   ```bash
   npx wrangler d1 execute typing-vocab-db --local --file=./schema.sql
   ```
2. 跑完 `--remote` 之后，实际查一下表结构，别只看有没有报错：
   ```bash
   npx wrangler d1 execute typing-vocab-db --remote --command "PRAGMA table_info(words);"
   ```
   确认新加的列真的出现在结果里。
3. `ALTER TABLE ADD COLUMN` 没有 `IF NOT EXISTS` 这种写法，老老实实写
   `ALTER TABLE words ADD COLUMN xxx ...` 就行；在已经有这列的库上重复跑同一个
   迁移文件会报 "duplicate column name"，这是正常现象（说明已经加过了），但
   因为上面那条"整文件事务"的坑，**不要**把这种"预期会重复报错"的迁移语句和
   还没跑过的新语句写进同一个文件——分开成独立文件才能保证互不连累。
